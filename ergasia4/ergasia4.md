## Εργαστηριακή Άσκηση 4 — Σημειώσεις και Απαιτήσεις

Προσωπικές μου σημειώσεις για την παρουσίαση της άσκησης από τον διδάσκοντα για την Εργαστηριακή Άσκηση 4. Χρησιμοποιήστε αυτό το έγγραφο ως αναφορά κατά την υλοποίηση του `client`.

### 1. Διαδικαστικά / Επικαιρότητα
- Εξέταση εργαστηρίου: **20 Μαΐου** (όπως αναφέρθηκε στην παρουσίαση).
- Ερωτήσεις/υποστήριξη: χρησιμοποιήστε το αντίστοιχο κανάλι του μαθήματος στο Slack.

### 2. Στόχος της Άσκησης
Υλοποίηση προγράμματος-πελάτη (`client`) που συνδέεται με τον server των διδασκόντων μέσω TCP/IP (sockets) και:
- Λαμβάνει μετρήσεις αισθητήρων (π.χ. θερμοκρασία).
- Υποβάλλει αιτήματα για άδειες μετακίνησης/εξόδου και διαχειρίζεται την πρωτοκόλληση επιβεβαίωσης.

### 3. Βασικά Θέματα / Θεωρητικό Υπόβαθρο
- Domain: Χρησιμοποιούμε το Internet domain (`AF_INET`) — όχι Unix-domain sockets.
- Τύπος σύνδεσης: Stream sockets (`SOCK_STREAM`) — πρωτόκολλο TCP.
- Τα sockets ταυτοποιούνται με `IP:PORT`.

### 4. Παράμετροι Εκτέλεσης
Το πρόγραμμα πρέπει να αποδέχεται τις παρακάτω παραμέτρους γραμμής εντολών:

```
./ask4 [--ip IP] [--port PORT] [--debug]
```

- Προεπιλεγμένες τιμές: `IP=147.102.75.201`, `PORT=4217`.
- Σημείωση: δεν επιτρέπεται η χρήση hostnames — μόνο διευθύνσεις IPv4 ως string.

### 5. Πρωτόκολλο Εφαρμογής — Υποστηριζόμενες Εντολές
Ο `client` πρέπει να διαβάζει εντολές από το τερματικό και να επικοινωνεί με τον server σύμφωνα με την παρακάτω διεπαφή:

- `help`
	- Εμφανίζει βοήθεια/συντομεύσεις.

- `exit`
	- Κλείνει ομαλά τη σύνδεση και τερματίζει τον client.

- `get`
	- Ζητάει από τον server την τρέχουσα μέτρηση αισθητήρα.
	- Ο server επιστρέφει συμβολοσειρά: `X YYY ZZZZ TTTTTTTT` όπου:
		- `X` = τύπος συμβάντος (π.χ. 0: boot, 2: interval, 4: motion)
		- `YYY` = επίπεδο φωτεινότητας
		- `ZZZZ` = θερμοκρασία (πρέπει να διαιρεθεί με 100 για πραγματική τιμή)
		- `TTTTTTTT` = UNIX timestamp (μετατρέψτε με `localtime()` / `strftime()`)

- `N name surname reason`
	- Υποβολή αίτησης άδειας (π.χ. `1 aggelos kolaitis dokimi`).
	- Πρωτόκολλο:
		1. Ο server επιστρέφει έναν κωδικό επιβεβαίωσης.
		2. Ο client διαβάζει τον κωδικό και τον στέλνει πίσω στον server.
		3. Ο server αποκρίνεται με `ACK` (επιτυχία) ή `try again` (αποτυχία — επαναπροσπάθεια).

### 6. Τεχνικές Απαιτήσεις / Περιορισμοί

- I/O Multiplexing: Το πρόγραμμα πρέπει να περιμένει ταυτόχρονα είσοδο από το τερματικό και από το socket — απαιτείται χρήση της system call `poll()`.
- Απαγορεύεται η χρήση `gethostbyname()` ή άλλων συναρτήσεων που επιλύουν hostnames.
	- Πρέπει να μετατρέψετε χειροκίνητα την IPv4 διεύθυνση σε bytes (π.χ. `sscanf()` ή `inet_pton()` εάν επιτρέπεται) και να γεμίσετε σωστά την `struct sockaddr_in`.

### 7. Σημεία Υλοποίησης / Προτάσεις
- Χρήση `poll()`:
	- Επιτηρήστε δύο fd: `STDIN_FILENO` και το socket fd. Όταν υπάρχουν δεδομένα από το socket, διαβάστε και χειριστείτε. Όταν υπάρχουν δεδομένα από το stdin, στείλτε κατάλληλο αίτημα στον server.

- Διαχείριση συμβολοσειρών και parsing:
	- Ελέγξτε τα όρια buffers, χρησιμοποιήστε `snprintf()` / `strtok_r()` και ελέγξτε την εγκυρότητα των τιμών πριν την επεξεργασία.

- Χρόνος αναμονής / επαναπροσπάθεια:
	- Ενσωματώστε timeout για το socket όταν περιμένετε επιβεβαίωση, και λογική επαναπροσπάθειας όταν ο server απαντά `try again`.

### 8. Παραδείγματα Εκτέλεσης

```
./ask4 --ip 147.102.75.201 --port 4217
> help
> get
< 2 512 2350 1653000000
	(θερμοκρασία = 2350 / 100 = 23.50 °C)
> 1 aggelos kolaitis dokimi
< CONFIRM_CODE_12345
> CONFIRM_CODE_12345
< ACK
```

### 9. Έλεγχος / Παράδοση
- Βεβαιωθείτε ότι ο client χειρίζεται σωστά:
	- `poll()` και πολλαπλά fd
	- parsing του μηνύματος `get`
	- πρωτόκολλο επιβεβαίωσης για αιτήματα άδειας
- Παραδώστε τον πηγαίο κώδικα και σύντομο `README` με οδηγίες εκτέλεσης και παραδείγματα.

### 10. Λογική του `client.c`
Ο κώδικας του client ακολουθεί ένα απλό αλλά σωστά δομημένο μοτίβο:

1. **Ανάγνωση παραμέτρων**
	- Διαβάζει `--ip`, `--port` και `--debug` από τη γραμμή εντολών.
	- Αν δεν δοθούν τιμές, χρησιμοποιεί τις προεπιλογές του εργαστηρίου.

2. **Δημιουργία socket και σύνδεση**
	- Δημιουργεί TCP socket με `socket(AF_INET, SOCK_STREAM, 0)`.
	- Συμπληρώνει τη δομή `sockaddr_in` με IP και port.
	- Η IP μετατρέπεται χειροκίνητα από string σε bytes, χωρίς `gethostbyname()`.
	- Καλεί `connect()` για να συνδεθεί στον server.

3. **Κεντρικός βρόχος με `poll()`**
	- Παρακολουθεί ταυτόχρονα δύο πηγές:
	  - το `STDIN_FILENO` για εντολές του χρήστη,
	  - το socket για απαντήσεις από τον server.
	- Έτσι ο client δεν «κολλάει» περιμένοντας μόνο μία από τις δύο πλευρές.

4. **Χειρισμός εντολών από το terminal**
	- `help`: εμφανίζει τις διαθέσιμες εντολές.
	- `exit`: τερματίζει ομαλά το πρόγραμμα.
	- οποιαδήποτε άλλη γραμμή: στέλνεται αυτούσια στον server.

5. **Χειρισμός δεδομένων από τον server**
	- Διαβάζει με `read()` την απάντηση του server.
	- Αν ο server κλείσει τη σύνδεση, ο client τερματίζει.
	- Αν το `debug` είναι ενεργό, τυπώνει επιπλέον πληροφορίες για αποστολές/λήψεις.

6. **Πού χρειάζεται ακόμη επέκταση**
	- Το `get` χρειάζεται parsing της μορφής `X YYY ZZZZ TTTTTTTT` και μετατροπή του timestamp.
	- Το `N name surname reason` χρειάζεται ειδικό χειρισμό επιβεβαίωσης (`ACK` / `try again`).
	- Δηλαδή ο τωρινός client έχει τη βασική υποδομή, αλλά το πρωτόκολλο εφαρμογής θέλει ακόμα εξειδίκευση.

### 11. Πιθανές Ερωτήσεις Καθηγητή
Αυτές είναι ερωτήσεις που είναι πιθανό να τεθούν προφορικά πάνω στον κώδικα ή στις αποφάσεις υλοποίησης:

- Γιατί χρησιμοποιείς `poll()` και όχι `select()`;
- Τι πρόβλημα λύνει το `poll()` στον συγκεκριμένο client;
- Ποια είναι η διαφορά μεταξύ `AF_INET` και `AF_UNIX`;
- Γιατί ο server/client πρέπει να χρησιμοποιούν `SOCK_STREAM`;
- Πώς μετατρέπεται μια IPv4 διεύθυνση από string σε network byte order χωρίς DNS;
- Τι κάνει η `htons()` και γιατί είναι απαραίτητη στο port;
- Τι θα συνέβαινε αν δεν έλεγες το `revents` μετά το `poll()`;
- Γιατί χρειάζεται έλεγχος επιστροφής σε `socket()`, `connect()`, `poll()`, `read()` και `write()`;
- Πώς θα χειριστείς το `get` μήνυμα και πώς θα μετατρέψεις τη θερμοκρασία από ακέραιο σε δεκαδικό;
- Τι σημαίνει ότι το `poll()` παρακολουθεί ταυτόχρονα stdin και socket;
- Πώς θα υποστηρίξεις το πρωτόκολλο επιβεβαίωσης για το αίτημα άδειας;

### 12. Σύντομη Απάντηση-Οδηγός για Προφορική Εξήγηση
- Ο client είναι event-driven: περιμένει γεγονότα από δύο πηγές με `poll()`.
- Η IP δεν επιλύεται με host lookup, αλλά μετατρέπεται χειροκίνητα σε bytes.
- Ο στόχος είναι να μη μπλοκάρει το πρόγραμμα όταν περιμένει τον χρήστη ή τον server.
- Το `debug` βοηθάει στον έλεγχο της ροής, χωρίς να αλλάζει τη λογική του πρωτοκόλλου.

### 13. Βοήθεια
Αν θέλεις, μπορώ να προσθέσω και μια μικρή ενότητα «τι να απαντήσεις» για καθεμία από τις παραπάνω ερωτήσεις, σε μορφή σύντομων bullet points.

### 14. Τι άλλαξε στον κώδικα

Οι βασικές αλλαγές που έγιναν στον `client.c` είναι οι εξής:

- Προστέθηκε χειροκίνητη μετατροπή IPv4 από string σε `struct in_addr`, χωρίς DNS resolution.
- Προστέθηκε parser για το `get` response, ώστε να εμφανίζονται σωστά event, θερμοκρασία, φωτεινότητα και timestamp.
- Προστέθηκε `read_server_message()` για ασφαλή ανάγνωση από το socket και αφαίρεση του newline.
- Προστέθηκε κεντρικό loop με `poll()` για ταυτόχρονη παρακολούθηση stdin και socket.
- Η αποστολή αιτήματος άδειας πέρασε σε state-driven ροή, ώστε οι αναγνώσεις του socket να γίνονται αποκλειστικά από το socket block.
- Προστέθηκε χειρισμός για `STATE_WAITING_GET_RESPONSE`, `STATE_WAITING_VERIFICATION` και `STATE_WAITING_FINAL_ACK`.
- Ο verification code επιστρέφεται πλέον με newline ώστε ο server να τον αναγνωρίζει σωστά ως ολοκληρωμένη γραμμή.
- Προστέθηκε debug mode για να φαίνονται οι αποστολές και οι λήψεις.

### 15. Τελική αξιολόγηση της αρχιτεκτονικής

Με τις αλλαγές αυτές, ο client έγινε πιο σωστός για event-driven I/O:

- Δεν μπλοκάρει μέσα στο terminal block όταν περιμένει τον server.
- Δεν διαβάζει το ίδιο socket από δύο διαφορετικά σημεία με ασύγχρονο τρόπο.
- Δεν κάνει τυφλό parsing σε μηνύματα που δεν είναι `get` responses.
- Η ροή του πρωτοκόλλου ελέγχεται πλέον από κατάσταση, όχι από έμμεσες υποθέσεις.

Αυτό κάνει την υλοποίηση πιο robust και πιο κοντά στη συμπεριφορά που ζητά η άσκηση.

## Ανάλυση Αρχιτεκτονικών Παγίδων στο Τρέχον Πρωτόκολλο

Ο κώδικας του client είναι εξαιρετικά καλογραμμένος, δομημένος, καθαρός και δείχνει πολύ καλή κατανόηση των system calls. Η απομόνωση των λειτουργιών σε συναρτήσεις, η χρήση της `sscanf()` με το `%c` και το bit-shifting στην IP είναι υποδειγματική.

Ωστόσο, η αρχιτεκτονική του περιέχει **τρεις μεγάλες δομικές παγίδες** που αφορούν τον τρόπο αλληλεπίδρασης της `poll()` με τις synchronous (blocking) κλήσεις `read/write` που εμφανίζονται ενδοτερικά στο block του terminal.

### Παγίδα 1: Το "Σπάσιμο" του Ασύγχρονου Μοντέλου της `poll()` (Nested Blocking)

#### Η Πρόκληση
Η `poll()` χρησιμοποιείται για να μην κολλάει (block) ποτέ το πρόγραμμα σε έναν file descriptor, επιτρέποντας στον client να «ακούει» ταυτόχρονα και το πληκτρολόγιο και το socket.

#### Τι Συμβαίνει στον Κώδικα
Μέσα στο block του terminal (`if(fds[0].revents & POLLIN)`), μόλις ανιχνεύσει αίτημα άδειας, εκτελούνται **χειροκίνητα και σειριακά** (synchronously) δύο διαδοχικές `read_server_message()` και μία `write()`:

```c
/* Μέσα στο block του terminal... */
int verification_read = read_server_message(sockfd, verification_buffer, sizeof(verification_buffer));
...
if (write(sockfd, verification_buffer, strlen(verification_buffer)) < 0) ...
...
int final_read = read_server_message(sockfd, verification_buffer, sizeof(verification_buffer));
```

#### Η Παγίδα
1. **Παράκαμψη της `poll()`**: Για όσο χρόνο το πρόγραμμα είναι "κλειδωμένο" μέσα σε αυτές τις εσωτερικές `read_server_message()`, η κεντρική `poll()` του `while(1)` **δεν εκτελείται**.

2. **Κίνδυνος Deadlock (Μπλοκάρισμα)**: Αν ο server καθυστερήσει να στείλει το verification code ή το τελικό `ACK`, ο client θα κολλήσει επ' αόριστον μέσα στο block του terminal. Αν εκείνη τη στιγμή ο χρήστης πληκτρολογήσει `exit`, ο client δεν θα το διαβάσει ποτέ, γιατί δεν "ακούει" το stdin.

#### Συνέπεια
**Ασφυξία του event loop**: Το πρόγραμμα γίνεται αδύνατο να αποκριθεί σε νέα γεγονότα μέχρι να ολοκληρωθούν όλες οι inline κλήσεις.

---

### Παγίδα 2: Η "Μάχη" για τα Δεδομένα του Socket (Race Condition μεταξύ Terminal & Socket Blocks)

#### Η Πρόκληση
Στον κώδικα υπάρχουν δύο διαφορετικά σημεία που διαβάζουν από το ίδιο socket (`sockfd`):
1. Το block του terminal (συγχρονισμένα, για την άδεια).
2. Το block του server (`if(fds[1].revents & POLLIN)`), το φυσικό σημείο εισόδου των δεδομένων του δικτύου.

#### Τι Συμβαίνει στον Κώδικα
Υποθέστε ότι στέλνετε το αίτημα άδειας `1 pantelis angelidis test`. Το πρόγραμμα βγαίνει από το block του terminal, το loop ξαναξεκινάει και η `poll()` αναφέρει ότι ήρθαν δεδομένα από τον server στο socket.

#### Η Παγίδα
Ποιο block θα "πιάσει" το verification code;

* Αν η `poll()` ξυπνήσει και εκτελεστεί το `if(fds[1].revents & POLLIN)` (το block του server) **πριν** ξαναμπεί στο block του terminal, η `read_server_message()` εκεί θα "καταπιεί" τον κωδικό επιβεβαίωσης.
* Το block του server θα δει ότι το string δεν είναι `get` response, θα τυπώσει `Server: [κωδικός]` και θα σταματήσει.
* Όταν στέλνετε την εσωτερική `read_server_message()` του block του terminal, τα δεδομένα θα έχουν ήδη διαβαστεί, και ο client θα κολλήσει περιμένοντας δεδομένα που δεν θα έρθουν ποτέ.

#### Συνέπεια
**Σιωπηλό deadlock**: Ο client φαίνεται να περιμένει κανονικά, αλλά εσωτερικά βρίσκεται σε ανίσχυρη κατάσταση· ο σωστός κωδικός έχει ήδη "κλαπεί" από ένα άλλο block.

---

### Παγίδα 3: Η Έλλειψη Κατάστασης (State Machine) και η "Τυφλή" Parsing του `get`

#### Η Πρόκληση
Στο block του server, η λογική λειτουργεί "τυφλά":

```c
if(!print_get_response(recv_buffer)) {
    printf("Server: %s\n", recv_buffer);
}
```

#### Τι Συμβαίνει στον Κώδικα
Η `print_get_response()` βασίζεται αποκλειστικά στο αν η `sscanf()` θα βρει 4 νούμερα. Ο client δεν "θυμάται" τι ζήτησε.

#### Η Παγίδα
1. Αν ο server στείλει ένα τυχαίο μήνυμα (π.χ. ένα error string ή ένα validation code) που για κάποιο περίεργο λόγο περιέχει 4 αριθμούς, η `print_get_response()` θα προσπαθήσει λανθασμένα να το μεταφράσει ως θερμοκρασία και event αισθητήρα.
2. Η σωστή αρχιτεκτονική για ασύγχρονα δίκτυα απαιτεί έναν **State Machine** (μιας μεταβλητής κατάστασης, π.χ. `enum State { IDLE, WAITING_GET, WAITING_VERIFICATION, WAITING_ACK };`).
3. Μόνο αν η κατάσταση είναι `WAITING_GET` θα έπρεπε να καλείται ο parser της `get`. Μόνο αν είναι `WAITING_VERIFICATION` θα έπρεπε να περιμένει τον κωδικό.

#### Συνέπεια
**Πρωτοκολλιακή διαστροφή**: Μηνύματα που δεν ήταν ποτέ προορισμένα να μεταφραστούν ως sensor data ενδέχεται να ερμηνευθούν λανθασμένα, προκαλώντας ψευδείς ανάγνωσες θερμοκρασίας ή φωτεινότητας.

### Διόρθωση

Για να γίνει ο κώδικας 100% εύρωστος (robust), **όλη η ανάγνωση του socket πρέπει να μεταφερθεί αποκλειστικά στο `if(fds[1].revents & POLLIN)`** και η ροή να κατευθύνεται με βάση την τρέχουσα κατάσταση (state) του client.

Η συνιστώμενη αρχιτεκτονική είναι:

1. **State Machine Enum**: Ορίστε τέσσερις καταστάσεις:
   - `STATE_IDLE`: Δεν περιμένουμε κάποιο ιδιαίτερο μήνυμα από τον server.
   - `STATE_WAITING_GET_RESPONSE`: Έχουμε στείλει `get` και περιμένουμε sensor data.
   - `STATE_WAITING_VERIFICATION`: Έχουμε στείλει αίτημα άδειας και περιμένουμε verification code.
   - `STATE_WAITING_FINAL_ACK`: Έχουμε στείλει τον κωδικό πίσω και περιμένουμε το τελικό ACK/try again.

2. **Terminal Block**: Στέλνει εντολές και θέτει την κατάλληλη state flag. **Δεν διαβάζει από το socket**.

3. **Socket Block**: Ενεργεί ως **κύριος ιδιοκτήτης** όλων των socket reads. Ερμηνεύει τα μηνύματα βάσει της τρέχουσας κατάστασης:
   - Αν `STATE_WAITING_GET_RESPONSE`: Καλεί `print_get_response()`.
   - Αν `STATE_WAITING_VERIFICATION`: Διαβάζει και αποστέλλει τον κωδικό.
   - Αν `STATE_WAITING_FINAL_ACK`: Διαβάζει το ACK/try again.

Αυτό εξασφαλίζει ότι:
- Δεν υπάρχει nested blocking (το loop ξαναρχίζει πάντα).
- Δεν υπάρχουν race conditions (ένα σημείο κατέχει όλες τις socket reads).
- Δεν υπάρχει τυφλή parsing (το state καθοδηγεί τη σωστή ερμηνεία).
